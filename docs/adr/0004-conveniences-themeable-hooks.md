# Conveniences expose themeable hooks; starters override only to deviate

The shared conveniences (`slidash.css`/`slidash.js` — chrome today; grid, presenter view, etc. later) read every themeable point through `var(--slidash-*, <fallback>)`. The embedded fallback _is_ the default, so the machinery is self-sufficient: it renders fully with no starter at all and never depends on a starter's tokens. A starter declares a `--slidash-*` variable **only when it wants to deviate** from the fallback — never to restate it. So `none` declares none (it accepts the neutrals), while a starter with a strong aesthetic (e.g. Indigo Ink) sets just the chrome/grid hooks it needs so the conveniences match its look.

We rejected declaring those defaults in each starter's `theme.css` "for coherence". Restating a fallback turns the starter into an _override_ of a value it only means to accept: `none` would ship a pinned copy of the machinery default, and as the conveniences evolve across releases that pin silently falls out of step with the real fallback. The hazard is in maintaining our template set (one fallback, restated across N starters), not in any already-scaffolded presentation — those are frozen and yours, never updated underneath you (ADR-0001), so this is not a runtime or backwards-compatibility concern. Duplication also blurs the deliberate boundary between the starter's content layer (`theme.css`: `--bg`, `--ink`, `--accent`, type, spacing) and the machinery's tuning surface (the `--slidash-*` namespace). The themeable contract is therefore one-way — the conveniences offer hooks with fallbacks, a starter opts in only to deviate — and the hooks are documented as a commented reference at the top of `slidash.css` — beside the machinery that reads them — rather than declared live for discoverability.
